## Bugs Fixed

### 🔴 Critical

**1. `DatabaseExportService::exportTableLaravel()` — Corrupted Laravel Migration Output**

**File:** `app/Services/DatabaseExportService.php` — line 237 (removed)

```php
// BEFORE (broken): appended after the closing `};` of every migration
$output_string .= $def . "\n?>"; /* the "includes" are still used multiple times ... */
```

Every Laravel migration export was syntactically broken. After the correct closing `};`, the service appended `$def` (the raw last column definition string from the loop, e.g. `$table->string('email')`) followed by a `?>` PHP closing tag. The result was a PHP file that would throw a parse error on `require` and produced garbage output.

**Fix:** Removed line 237 entirely. The `};` line above it is the correct terminator.

---

### 🟠 Security Issues

**2. `ProjectController::edit()`, `update()`, `destroy()` — Missing Ownership Check**

**File:** `app/Http/Controllers/ProjectController.php`

These three methods accepted a `Project $project` via route-model binding but had **no authorization check**. Any authenticated user who knew (or guessed) another user's project slug could:
- `GET /p/{slug}/edit` — view the project edit form
- `PUT /p/{slug}` — rename/update a project they don't own
- `DELETE /p/{slug}` — permanently delete another user's project

Note: `SchemaController` and `ProjectController::index()` correctly had `abort_if($project->owner_id !== auth()->id(), 403)`, making this an inconsistency that was easy to miss.

**Fix:** Added `abort_if($project->owner_id !== auth()->id(), 403)` as the first line of each of the three methods.

**3. `AuthController::signup()` — No Minimum Password Length**

**File:** `app/Http/Controllers/AuthController.php`

```php
// BEFORE
'password' => 'required',

// AFTER
'password' => 'required|string|min:8',
```

The signup route accepted passwords of any length, including a single character or an empty string passed via a crafted request. This violates OWASP minimum password hygiene standards.

**Fix:** Added `string|min:8` to the password validation rule.

---

### 🟡 Minor Issues

**4. `SchemaTable` Model — Missing `SoftDeletes` Trait**

**File:** `app/Models/SchemaTable.php`  
**New migration:** `database/migrations/2026_07_18_000000_add_soft_deletes_to_schema_tables.php`

`SchemaDatabase` and `SchemaColumn` both use `SoftDeletes`, but `SchemaTable` did not. This created an asymmetry:
- Soft-deleting a column correctly hides it from all queries.
- But soft-deleting a database would not propagate to tables — tables under a deleted database remained queryable and visible in the table list.

**Fix:** Added `use SoftDeletes` to `SchemaTable` and created a migration to add `deleted_at` to `schema_tables`.

**5. Dashboard — Raw Debug `"NEW"` Link**

**File:** `resources/views/pages/dashboard.blade.php`

```html
<!-- BEFORE -->
<a href="{{ route('new') }}">"NEW"</a>
```

The dashboard had a leftover debug anchor with the literal text `"NEW"` (including the quotation marks). No other UI context was present — the dashboard was effectively a blank page for users.

**Fix:** Replaced with a proper "Quick Start" section card containing styled `btn-primary` and `btn-secondary` action buttons.

## Architecture & Code Quality

### ✅ What's Good

| Area | Observation |
|---|---|
| **Route structure** | Excellent. Literal sub-routes correctly precede wildcards (`_tables`, `export` before `{table:name}`). Named routes are consistently used throughout views. |
| **Model design** | Good use of ULIDs, SoftDeletes (post-fix), eager loading with closures, and typed relation methods. |
| **Authorization pattern** | Consistent and explicit. Every SchemaController entry point has a project ownership check + parent-child relationship checks. No Policy/Gate overhead needed at this scale. |
| **Service layer** | Clean separation. `DatabaseExportService` and `DatabaseImportService` are genuinely service objects — no DB access leaking into controllers beyond what is necessary. |
| **JS architecture** | Impressively clean single-file approach. State is a plain JS object; `isDirty()` diffs via JSON.stringify; autosave countdown is well-implemented. The `buildPayload()` / `readColFromRow()` / `syncColumnsOrderFromDOM()` separation is good. |
| **Import service** | The additive-only import strategy (never destructive) is the right default. FK resolution is honest about its limitations (best-effort). |
| **Test suite** | 48 tests / 103 assertions covering auth, project CRUD, schema CRUD, JSON API, and export. All tests use `RefreshDatabase` and a reusable `loginUser()` helper. |

### ⚠️ Observations (Not Bugs, But Worth Noting)

**A. `Project::$schema` is a Mutable Public Instance Property**

```php
class Project extends Model
{
    public $schema = ['name' => 'projects', 'base_route' => 'projects', 'columns' => [...]]
```

This config-as-property pattern is a code smell. Because it is a non-static public instance property it is technically mass-assignable (though `$fillable` prevents it in practice), can be overwritten at runtime, and every `new Project()` call (done 4+ times in `ProjectController`) allocates this array.

**Recommendation:** Convert to `public static array $schema = [...]` or a dedicated readonly method.

**B. `DatabaseExportService::exportDatabase()` Duplicates the Auth Check**

The service re-checks `$project->owner_id !== Auth::id()` even though the controller already did `abort_if(...)`. The double-check is harmless (defence-in-depth) but injecting `Auth::` into a service class is an anti-pattern — services should not be aware of the request context.

**Recommendation:** Remove the auth check from the service; keep it exclusively in the controller.

**C. `SchemaController::updateTable()` — `$col->fresh()` Inside a Loop**

```php
$savedColumns[] = $col->fresh(['referencedTable']);
```

`fresh()` re-fetches each model from the database individually (N+1 queries for N columns). Since `$table->load('columns.referencedTable')` is called immediately after the loop anyway, `$savedColumns` could be built from `$table->columns` instead.

**D. SQL Import Regex — Edge Case With `)` in DEFAULT Values**

```php
preg_match_all('/CREATE\s+TABLE\s+`?(\w+)`?\s*\((.*?)\)\s*;/is', $sql, ...);
```

The `.*?` in `DOTALL` mode handles newlines correctly but can fail on SQL that has a `)` inside a string literal in a `DEFAULT` clause (e.g. `DEFAULT '(none)'`). This is an edge case but worth documenting.

**E. `navbar.blade.php` Is an Empty Placeholder**

```html
<nav>
    
</nav>
```

Users currently have no persistent site-wide navigation. Each page provides its own breadcrumb but there is no header with logo, user identity, or logout shortcut. Fine for an early-stage app but worth addressing before shipping.
